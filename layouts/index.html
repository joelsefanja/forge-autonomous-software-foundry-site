03 / Proof

Most software risk appears before the build starts.

FORGE checks whether the problem is specific, repeated and worth solving. If a maintained tool or integration solves it safely, building is not the first answer.

Problem

A good-looking demo is not a product.

A demo can look good and still be impossible to sell, support, audit or hand over. FORGE treats demand proof, quality checks and owner documentation as part of the product.

Method

Every step asks whether to build, simplify or stop.

01 Collect repeated pain, workarounds and spending signals.
02 Narrow the offer to one buyer, one job and one first promise.
03 Check whether an existing tool, API or workflow can solve it first.
04 Build the smallest version only when reuse is not enough.
04 / Process

From customer signal to reviewable software.

Each step produces a concrete artifact, so founders, builders and buyers can see what was learned, what changed and why the product is ready to build or stop.

01
Find the pain
Find repeated customer pain in reviews, forums, job posts, issue trackers, support gaps, niche communities and existing product workflows.
02
Check the buyer
Check who pays, what they use now, what is missing, what risks exist and whether the first version can stay narrow.
03
Capture the workflow
Record discovery flows, screenshots, videos, raw HTML/text and optional llms.txt context while respecting robots.txt by default.
04
Ship the reviewable product
Create the first useful version with tests, UI audit evidence, accessibility checks, security review, release notes and deployment setup.
05
Prepare go-to-market
Prepare the landing page, pricing, onboarding, payment path, follow-up automations and support model.
06
Transfer ownership
Package docs, risks, operating notes, credentials checklist and transfer steps.

06 / Fit

For teams that need confidence before build speed.

FORGE fits when the first product must be narrow, explainable and transferable, and when reuse should be checked before custom build work.

Best fit
01 Founders who keep hearing the same customer problem and need to know what to build first.
02 Agencies turning repeatable client work into a productized software offer.
03 Operators replacing fragile manual workflows with owned software.
04 Product buyers who need proof, risks and handover context before approving a build.
Not a fit

Not for broad platform bets.

FORGE works best when the first release can serve one buyer, one job and one measurable outcome. Broad platforms should come later, if customers prove they are needed.

07 / Questions

The buyer questions FORGE should answer early.

The process is designed to reduce uncertainty before a build becomes expensive or hard to transfer.

Can I stop after validation?

Yes. A stop decision is a valid output when the evidence is not strong enough.

Who owns the deliverables?

The product owner owns the code, docs and package unless agreed otherwise.

What if I already have an MVP?

FORGE can review the customer demand, version-one plan, quality notes and handover readiness before more build work.

Do you replace my technical team?

No. FORGE uses agents as bounded execution workers and leaves a reviewable package your team can continue.
